# 多店模式 — 规格备忘

> 本文档汇总单店改为多店（路径型租户）的产品与实现共识，供后续开发对照。  
> 实现时以当时最新数据模型与路由为准，本文不替代代码。

**更新**：2026-05-05（含 slug / 订阅口径 / 审计结论）

---

## 1. 目标与部署

- **一套 Web App、同一主域名**（例：`www.lztechserve.com`），**仅增加路径路由**承载多店，不拆多套部署实例。
- **租户识别**：路径第一段为店铺 **slug**，例如 `/{slug}/customer/...`、`/{slug}/cashier/...`、`/{slug}/admin/...`。
- **数据隔离**：业务数据按 **店（store）** 维度隔离，互不干扰。

---

## 2. 首页与登录

| 项 | 约定 |
|----|------|
| **`/`** | **公司介绍**；**超级管理员入口**的具体交互与路由**后议**（可先占位）。 |
| **登录** | **每店独立登录页**：`/{slug}/login`。 |

---

## 3. 店铺标识与名称

- **slug（URL 段）**：创建时确定；**创建后不允许修改**（二维码、收藏、外链均依赖稳定 slug）。
- **店铺展示名称**：**可更改**（后台编辑，不影响 URL）。
- **内部主键**：实现上使用稳定 `storeId`（如 ObjectId）关联数据；slug 仅作解析入口，需唯一索引。

---

## 4. 角色与平台管理员

- **`admins` 角色扩展**：在现有 `owner`、`cashier` 基础上增加 **`platform_owner`**（平台超级管理员）。
- **代操作**：`platform_owner` **允许**进入任意店铺并代操作（能力对齐该店业务后台/收银；具体 UI 是否区分「代操作」模式可迭代）。
- **审计日志（v1 要求）**：对 `platform_owner` 的代操作/跨店访问记录 **谁、何时、目标店铺、关键操作**（写操作至少），便于追责与合规。

---

## 5. 订阅与可用期

| 项 | 约定 |
|----|------|
| **起算** | **创建店铺即刻**起算可用期。 |
| **结束日 vs 天数** | **以结束日为最终依据**；**天数仅作快捷输入**（用于自动推算结束日，可覆盖手填）。 |
| **配置** | 支持 **任意天数** + 后台 **手填结束日**（二者逻辑统一在「结束日」上）。 |
| **到期后** | 数据 **保留 3 个月**；支持 **一键续费/延期**；延期后 **按新结束日立即生效**（服务恢复规则与前端提示可随实现细化）。 |
| **到期行为（默认）** | 若未另定：顾客端 **禁止下单**，可只读提示；收银/后台按锁定策略执行（细则可与产品再对齐）。 |

---

## 6. 按店独立的配置与业务

- **Stripe**：**每店独立**商户与密钥；在 **该店管理员后台** 自行配置。
- **会员 / 集点**（若实施）：**按店独立**（余额、印花不跨店）。
- **小票 / 公司信息 / 税号等**：**每店独立配置**。

---

## 7. 实时通道（Socket.IO）

- 实时订单等通道按 **店 + 房间（room）** 隔离，连接与订阅前 **校验店铺上下文**，避免收银 **串店**。

---

## 8. 单店数据迁移（概念）

- 现有生产数据归入 **默认店铺**（固定 slug，迁移时一次性绑定）；旧无 slug 的 URL 是否兼容重定向，单独在迁移方案中列出。

---

## 9. 与会员规格的关系

- 会员模块见 `docs/membership-module-spec.md`；多店下会员 **scope 为单店**，不与跨店通用冲突。
